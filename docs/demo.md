# Public Demo

EngMetrics AI ships with a fully interactive demo dashboard that requires zero credentials and no configuration. This page explains what the demo shows and how the synthetic data maps to a real team scenario.

---

## Try it now

**Online (no install required):**
[engmetrics.ai/demo.html](https://engmetrics.ai/demo.html) — fully interactive, no login.

**Locally (from a clone):**

```bash
# Option 1 – run the generator
python -m ai_engineering_metrics analyze --epic DEMO-1 --mock --output ./generated/demo.html
open ./generated/demo.html   # or: start ./generated/demo.html on Windows

# Option 2 – open the pre-committed snapshot
open examples/demo-dashboard.html

# Option 3 – Docker (no Python needed)
docker compose up
open generated/demo.html
```

---

## What you are looking at

The demo renders a realistic but fully synthetic engineering scenario:

| Field | Value |
|---|---|
| **Epic** | `DEMO-1` — "AI-Assisted Feature Rollout" |
| **Team size** | 3 engineers |
| **Duration** | 2 sprints (~4 weeks) |
| **Stories** | 5 user stories, 15–34 story points each |
| **Pull requests** | 8 PRs (6 merged, 1 open, 1 reverted) |
| **AI usage** | 4 of 5 stories have AI token data |

**All data is 100% synthetic.** No real organisation, repository, email address, or token cost appears in the demo. It is safe to share and screenshot.

---

## KPI cards explained

| Card | What it shows |
|---|---|
| **Lead Time** | Days from first commit to last merge — lower is better |
| **Deployment Frequency** | Merged PRs per working day — a proxy for deploys |
| **Change Failure Rate** | % of PRs that were reverts or hotfixes |
| **MTTR** | Average hours for a hotfix PR to merge after opening |
| **AI Adoption Rate** | % of stories where AI tools were used |
| **Hours Saved** | Estimated developer-hours saved via AI assistance |
| **Token Cost** | Estimated spend on AI tokens across the epic |
| **Rework Rate** | % of PRs that had commits pushed after a review comment |

See [docs/metrics.md](metrics.md) for the full definition, calculation method, and limitations of each metric.

---

## DORA performance bands

The demo shows a team performing in the **High** band across most DORA metrics. The revert PR pushes the Change Failure Rate into **Medium**. This is intentional — it demonstrates how the dashboard highlights real delivery signals, not just green numbers.

---

## Per-PR evaluation popup

Click any row in the **Pull Requests** table to open the evaluation popup. It shows:

- **Risk score** (0–100) — blends complexity, rework, and AI dependency signals.
- **Code quality heuristics** — diff-based proxies for complexity, code smells, and test coverage.
- **Rework signal** — commits pushed after the first review comment.
- **Improvement tooltips** — specific, actionable suggestions for each signal.
- **GitHub link** — in a real run this opens the PR directly; in demo mode it is disabled.

---

## What the demo does NOT show

- Real team data, names, repositories, or Jira URLs.
- Static-analysis scores (SonarQube/CodeClimate) — the demo uses heuristic proxies.
- Jira epic hierarchy beyond the `DEMO-1` epic.
- Multi-epic portfolio rollups (on the [roadmap](../ROADMAP.md)).

## Regenerating the demo

The `examples/demo-dashboard.html` committed to the repo is generated by:

```bash
python -m ai_engineering_metrics analyze --epic DEMO-1 --mock --output examples/demo-dashboard.html
```

Only regenerate it if the dashboard template or mock data changes. Commit the result with a message like `chore: regenerate demo dashboard`.

> **Never commit a dashboard generated from real data.** The `reports/` and `generated/` directories are git-ignored for this reason.
